chat bom dia, acabei de criar um projeto laravel com, composer create-project --prefer-dist laravel/laravel nome-do-projeto. Desejo que me instrua para versionar esse projeto com CI/CD pipiline
O ChatGPT disse:

Bom dia! Que legal que você criou um projeto Laravel! Para versionar seu projeto com CI/CD (Integração Contínua / Entrega Contínua), você pode seguir alguns passos principais, que envolvem:

  1. Configuração de Git

  2. Escolher uma ferramenta de CI/CD

  3. Configuração do Pipeline

  4. Deploy automatizado

Vou te guiar em cada etapa. Aqui vai o processo detalhado:

1. Configuração de Git

Primeiro, você precisa inicializar um repositório Git no seu projeto, caso ainda não tenha feito isso:

bash
cd nome-do-projeto
git init
git add .
git commit -m "Primeiro commit"

Depois, você pode criar um repositório remoto em plataformas como GitHub, GitLab ou Bitbucket, e vincular seu repositório local ao remoto:

bash
git remote add origin https://github.com/seu-usuario/nome-do-projeto.git
git push -u origin master

2. Escolher uma ferramenta de CI/CD

Agora que seu projeto está versionado, você pode escolher uma ferramenta de CI/CD. Algumas das mais comuns são:

  • GitHub Actions (para projetos hospedados no GitHub)

  • GitLab CI/CD (para projetos hospedados no GitLab)

  • CircleCI

  • Jenkins (mais robusto, mas requer mais configuração)

  • Travis CI

Abaixo, vou te mostrar como configurar com o GitHub Actions, já que é bastante simples de configurar com GitHub.

3. Configuração do Pipeline com GitHub Actions

a) Criando o arquivo de workflow

  1. Crie a pasta .github/workflows no seu projeto.

  2. Dentro dessa pasta, crie um arquivo ci.yml (ou qualquer nome de sua preferência).

O conteúdo do arquivo ci.yml pode ser algo como:

yaml
name: Laravel CI/CD Pipeline

on:
  push:
    branches:
      - master
  pull_request:
    branches:
      - master

jobs:
  build:
    runs-on: ubuntu-latest
    
    steps:
      # Passo 1: Checar o código
      - name: Check out the code
        uses: actions/checkout@v2

      # Passo 2: Configurar PHP
      - name: Set up PHP
        uses: shivammathur/setup-php@v2
        with:
          php-version: '8.1'
          extensions: mbstring, bcmath, zip
          coverage: xdebug

      # Passo 3: Instalar dependências
      - name: Install dependencies
        run: |
          composer install --no-interaction --prefer-dist

      # Passo 4: Rodar os testes
      - name: Run tests
        run: |
          php artisan migrate --env=testing
          php artisan test
          
      # Passo 5: Build do projeto (opcional)
      - name: Build project (opcional)
        run: npm install && npm run dev

b) Explicando os passos

  1. Verificação do código: A ação actions/checkout@v2 faz o checkout do código mais recente do repositório.

  2. Configuração do PHP: A ação shivammathur/setup-php@v2 configura a versão do PHP e as extensões necessárias para o seu projeto Laravel. Você pode ajustar a versão de PHP conforme necessário.

  3. Instalação de dependências: A linha composer install vai instalar todas as dependências do Laravel.

  4. Execução de testes: O comando php artisan test roda os testes do Laravel para garantir que tudo esteja funcionando corretamente. Antes disso, é feito o migrate para rodar as migrações no banco de dados de teste.

c) Realizando o deploy (opcional)

Se você deseja adicionar o deploy automatizado, pode configurar um passo após a execução dos testes para fazer o deploy para um servidor, por exemplo. Para isso, você pode adicionar um passo no arquivo ci.yml como:

yaml
      - name: Deploy to production server
        env:
          SSH_PRIVATE_KEY: ${{ secrets.SSH_PRIVATE_KEY }}
        run: |
          echo "$SSH_PRIVATE_KEY" | ssh-add -
          ssh user@your-server 'cd /caminho/do/projeto && git pull && composer install && php artisan migrate'

Você pode configurar a chave SSH no GitHub, em Settings > Secrets.

4. Acompanhamento e melhorias

  • Notificações: Você pode configurar notificações no Slack ou por e-mail para acompanhar o andamento das execuções do pipeline.

  • Ambientes: Se você tiver ambientes diferentes (desenvolvimento, staging, produção), pode criar workflows diferentes para cada um e garantir que o deploy seja feito corretamente para cada ambiente.

Esse foi o passo básico de configuração com GitHub Actions. Caso queira usar outra ferramenta de CI/CD, posso te ajudar com mais detalhes.

Sim, exatamente! O script de criar a pasta .github/workflows deve ser feito no seu projeto local. Como você está usando o GitHub, sim, o GitHub Actions é a ferramenta de CI/CD ideal para o seu caso.

Como funciona o fluxo com GitHub Actions:

  1. No seu projeto local, você cria a pasta .github/workflows e o arquivo de configuração do workflow (ex: ci.yml).

  2. Depois de adicionar o arquivo de workflow e comitar essa alteração no seu repositório local, você faz o push para o GitHub:

bash
git add .github/workflows/ci.yml
git commit -m "Configuração inicial do GitHub Actions"
git push origin master

  1. Quando você fizer o push, o GitHub automaticamente vai detectar o arquivo de workflow e iniciar a execução do pipeline conforme configurado no arquivo ci.yml. Ou seja, ele vai rodar os testes e outras etapas definidas.

  2. A cada push para o repositório ou pull request (como configurado no workflow), o GitHub Actions vai rodar o processo que você definiu, como a instalação das dependências e execução dos testes.

O que você deve verificar no GitHub:

  • Após o primeiro push, acesse seu repositório no GitHub, clique em Actions (uma nova aba que vai aparecer no topo do seu repositório).

  • Você verá o status do pipeline de CI/CD que você configurou.

  • Se tudo estiver correto, o GitHub vai começar a rodar as ações configuradas no seu workflow, e você verá as execuções detalhadas com logs para acompanhar o que aconteceu durante o processo.

Resumo:

  • A configuração do GitHub Actions é feita no seu projeto local.

  • Depois de configurar e fazer o push, você vai acompanhar os status de execução diretamente no GitHub, na aba Actions.
